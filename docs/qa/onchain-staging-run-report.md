## ON-1 — Gasless attestation broadcast is broken (Major, environmental)

**Symptom:** On both staging (deployed) and this PR's local build, creating a project with a Privy social-login smart account on OP Sepolia: the embedded wallet signs (`secp256k1_sign`) and the ZeroDev smart-account client contacts the OP Sepolia bundler (`eth_chainId`, `eth_supportedEntryPoints`, `eth_blockNumber`, `eth_getTransactionCount`, `eth_estimateGas` — all 200) — **but the flow never calls `eth_estimateUserOperationGas` or `eth_sendUserOperation`.** No sponsored userOp is broadcast, no tx hash, nothing reaches the indexer's attestation listener, and the UI throws "There was an error creating … project." No funds were ever demanded (so it is **not** a gas/funds block).

**Evidence:**
- Staging: reproduced 3×; project never created (`/v2/projects?q=…` → totalCount 0; slug stays available). Screenshots `stg-prj01-fail.png`, `stg-creating.png`.
- Local A/B (this branch, same ZeroDev project `f895c488-…`, same backend): **reproduced identically** — 309-entry HAR contains no `*UserOperation*` method; project not created (`qa-test-ab32` still free). Screenshots `ab-ready-submit.png`, `ab-final-state.png`.

**Attribution:** Because the PR branch behaves identically to deployed staging against the same ZeroDev/backend config, the defect is **environmental — ZeroDev OP Sepolia paymaster/bundler side** (the smart-account client probes the bundler but the sponsorship/userOp-construction step throws before estimate/send). Not introduced by this PR.

**For the dev team (suggested root-cause directions):**
- Check the ZeroDev project's **gas/sponsorship policy and credits for OP Sepolia (11155420)** — a rejected/expired sponsorship would abort before `eth_sendUserOperation`.
- Check the **EIP-7702** path (`useEIP7702: true` in `utilities/gasless/config.ts`) and the embedded-account → ethers-signer conversion (`utilities/gasless/*`, `privy-signer`) for an exception thrown during userOp construction.
- The error toast is generic — add error detail/logging at the gasless send step so this isn't silent.
- Positive: on failure the create dialog correctly preserves all entered data (OCH-PRJ-02 behavior).
