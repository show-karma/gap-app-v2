# gap-app-v2 — Manual Regression & Validation Plan

This plan certifies gap-app-v2 after the quality-baseline refactor (dead-code removal, type tightening, react-doctor fixes, lint cleanup). It is a working, hand-executed checklist covering public discovery, authentication, project profiles, community management, the funding platform, applications, grants/milestones, impact, donations, payouts, KYC, and AI features. Run every case in a real browser and record the outcome before sign-off.

> **Last generated:** Generated by QA workflow.

---

## How to use this plan

1. **Browser**: Execute every case by hand in a real browser (Chrome recommended; repeat the responsive sweep on a 375px mobile viewport and the a11y sweep with keyboard + a screen reader). Do not rely on automated runners — this is a human certification pass.
2. **Record results per ID**: Mark each test-case ID as **Pass**, **Fail**, or **Blocked** (Blocked = cannot execute due to missing seed data, environment, or an upstream failure). Note the actual behavior next to any Fail/Blocked.
3. **Log the build**: Record the build/commit SHA (and tenant/host if whitelabel) at the top of your run sheet so failures are traceable to a specific deploy.
4. **File bugs by case ID**: For every Fail, open an issue that references the case ID (e.g. "PUB-011 — navbar flashes Sign in during hydration"). Reuse the exact Steps/Expected result from the row.
5. **Reuse the global checklist**: The [Cross-cutting / global checklist](#cross-cutting--global-checklist) (`GLB-*`) encodes app-wide rules (triad, Web3, pluralization, destructive confirms). Later domain cases reference it instead of repeating it; run the `GLB-*` cases against the busiest route of each domain.

### "Path type" column legend

| Path type | Meaning |
|-----------|---------|
| **Happy** | Primary success path with valid inputs and an authorized user. |
| **Alternate** | A valid but secondary variation (optional fields, deep-link entry, toggles, redirects). |
| **Negative** | Invalid input / validation must block the action. |
| **Boundary** | Edge of a range (0, exactly 1, max, very long/unicode) — focuses on pluralization and layout. |
| **Empty** | Zero-item dataset — a purpose-built empty state must render (never blank). |
| **Loading** | Slow/throttled network — a skeleton/spinner must render before data. |
| **Error** | API/network failure — a visible error state with retry must render (never blank/infinite spinner). |
| **Permission / RBAC** | Role-gated visibility — control must be shown / hidden / blocked per role. |
| **Web3** | Wallet failure modes (disconnect, wrong network, reject signature, insufficient gas, pending tx, Privy↔Wagmi race). |
| **Destructive** | Delete / revoke / regenerate / pause — must show a confirmation dialog. |
| **A11y** | Keyboard operability, focus trap, visible focus, accessible names, SR labels. |
| **Responsive / Mobile** | Layout at 375px / 768px without overflow; tap targets ≥44px. |
| **SSR / Hydration** | Server markup matches client; no hydration-mismatch warnings; JSON-LD present where expected. |
| **Multi-tenant / Whitelabel** | Tenant branding applied; data scoped to the tenant; no cross-tenant leakage. |
| **Note** | An assertion about expected absence/state of a feature (raise as needs-info if violated). |

## Test environment & accounts

### Local environment
- **Run locally**: `cd gap-app-v2 && pnpm run dev`, app served at `http://localhost:3000`.
- **Feature flags / env**: `NEXT_PUBLIC_E2E_AUTH_BYPASS` must be `false`/unset for real auth runs. Whitelabel runs require a tenant subdomain/host that resolves `isWhitelabel=true`.
- **Devices**: desktop (≥1280px), tablet (~768px), mobile (~375px); keyboard-only; a screen reader (VoiceOver/NVDA) for the a11y sweep.

### Authentication (Privy test account)
- Log in through Privy using the **test-account fixed-OTP convention**: email login with a deterministic OTP for the QA fixture account. The qa-executor skill reads these from environment variables — use **`$QA_TEST_EMAIL`** for the login email and **`$QA_TEST_OTP`** for the fixed one-time code. Do not hardcode real credentials.
- Also exercise social logins (Google, Farcaster) and an injected/external wallet (MetaMask/Rainbow) for the Web3 and identity-resolution cases.

### Required roles / accounts
RBAC roles are resolved server-side by the `GET permissions` endpoint (`authorizationService.getPermissions`) keyed on `communityId`/`programId`, **not** by client-only flags. Hierarchy: `NONE(-1) < GUEST(0) < APPLICANT(1) < PROGRAM_REVIEWER(2) < MILESTONE_REVIEWER(3) < PROGRAM_ADMIN(4) < PROGRAM_CREATOR(5) < COMMUNITY_ADMIN(6) < REGISTRY_ADMIN(7) < SUPER_ADMIN(8)`.

| Alias / role | Capability | How granted |
|--------------|-----------|-------------|
| `ANON` / guest | not logged in (incognito) | logout / fresh browser |
| `wallet-user` | authenticated via external wallet; ≥1 owned project | Privy + MetaMask connect |
| `social-user` | authenticated via email AND via Google/Farcaster (no wallet address) | Privy email + social |
| `empty-user` | authenticated, zero projects/donations/connections/applications | fresh account |
| `W-APPLICANT` | `APPLICANT` — owns own applications only | submitted an application |
| `W-PREVIEWER` | `PROGRAM_REVIEWER` on Program P1 | reviewer assignment (PROGRAM) |
| `W-MREVIEWER` | `MILESTONE_REVIEWER` on Program P1 | reviewer assignment (MILESTONE) |
| `W-PADMIN` | `PROGRAM_ADMIN` (≥ create program / edit builder / approve-reject) | program admin |
| `W-CREATOR` | `PROGRAM_CREATOR` / `isProgramCreator` | created a program in funding-map |
| `W-CADMIN` | `COMMUNITY_ADMIN` of Community A only | community admin attestation on A |
| `W-REGISTRY` | `REGISTRY_ADMIN` / `isRegistryAdmin` (Karma Allo profile member) | registry admin membership |
| `W-STAFF` | `SUPER_ADMIN` ("Staff"), wildcard `*`, `isOwner` for /admin & /super-admin | staff allowlist / owner store |
| `W-FAUCET` | Faucet admin (`useFaucetAdmin` → isAdmin) — independent of community RBAC | faucet admin allowlist |
| `W-NONE` | authenticated wallet with NO roles on A → resolves to GUEST/NONE | any fresh wallet |

### Required networks / chains
- A wallet configured with at least one `gapSupportedNetworks` chain (e.g. Optimism) **plus** one UNSUPPORTED chain (to force the wrong-network/switch path).
- One wallet **drained of gas** (insufficient-funds path) and one wallet able to reject signatures/txs.

### Seed data
- **Community A** (`community-a`, multi-program): ≥2 funding programs (P1 fully configured + enabled; P2 with NO form / disabled), ≥1 program with 0 applications, ≥1 program with >25 applications (pagination), categories, tracks, projects, indicators.
- **Community B** (`community-b`) where W-CADMIN is NOT an admin (cross-tenant isolation).
- A community with **exactly one** program and **exactly one** application (singular/plural checks); a community with **zero** projects/updates/financials/portfolio-reports (empty states).
- Applications across all statuses: `pending`, `under_review`, `revision_requested`, `resubmitted`, `approved`, `rejected`, `draft`; ≥1 approved (Milestones tab) and one with an AI evaluation + one without.
- A **project** with grants, team (≥2 members), impact, updates; a project with exactly **1** team member and **0** grants; a **grant** with milestones in mixed states (pending/completed/verified), a grant with 0 milestones, a completed grant, and a grant tied to an invoice-required funding program.
- An **access-code-gated** program, a **closed/disabled** program, a saved **draft** application.
- A bad/non-existent community id, project id, programId, applicationId, and report runDate (404 coverage).
- A **whitelabel tenant** domain pointing at Community A.

### Needs a real wallet / on-chain funds (flag these for environment readiness)
- Project creation attestation, grant/milestone create/edit/delete (on-chain revoke), milestone/grant completion, completion verification, grant-completion revoke (multiRevoke), endorsements, donation checkout (ERC20 approve + Permit2 batch), Safe disbursement proposal, Hedgey claim-funds (direct / delegated / via-Safe), chain payout address, faucet emergency controls, `/seeds/fund` contribution, `/api/sponsored-txn` gasless relay. These require a funded test wallet (and, for Safe flows, a deployed multisig + one non-deployed).

## Cross-cutting / global checklist

> Run these `GLB-*` cases against the busiest routes of every domain (grants, milestones, donations, applications list, AI reports, manage/admin views). They encode the project's hard rules: components must **never silently render nothing** (always a loading skeleton OR empty state OR error state with retry), pluralization must be correct, zero-count blocks hide entirely, mutations are optimistic with correct cache invalidation, destructive actions confirm, and Web3 failure modes are handled gracefully. Later cases reference these by ID rather than repeating them.

| ID | Path type | Precondition | Steps | Expected result |
|----|-----------|--------------|-------|-----------------|
| GLB-001 | Loading | Any data-fetching route | 1. Throttle to Slow 3G. 2. Hard-navigate to the route. | A loading skeleton/spinner renders during fetch; never a blank white area while pending. |
| GLB-002 | Empty | Account/community with zero of the relevant entity | 1. Navigate to the route. | A purpose-built empty state with copy + (where relevant) a CTA renders; component does NOT return `null`/blank. |
| GLB-003 | Error | Any data-fetching route | 1. Block the underlying API request (or return 500). 2. Load the route. | A visible error state with a retry affordance renders; clicking retry re-issues the request; no blank screen, no infinite spinner. |
| GLB-004 | Boundary (pluralization) | Lists with counts (grants, milestones, cart items, applications, donations, recipients, keys, tasks) | 1. Reach exactly 1, then 2, then 0 items. | Copy uses `pluralize`: "1 milestone" / "2 milestones" — never "1 milestones". Zero reads "0 milestones" or hides (GLB-005). No "0 apply"/"1 teams". |
| GLB-005 | Boundary (zero-count hide) | Any summary block tied to a count (e.g. "Closing this week — N", pending verifications, pending badges, cart badge, dashboard summaries) | 1. Drive the count to 0. | The entire count-tied block is hidden (not rendered as "0 …"). |
| GLB-006 | Mutation | Any mutation (cart add, create/edit/delete milestone, save KYC config, submit application, set payout address, assign reviewer) | 1. Perform the mutation. 2. Observe UI before/after server confirms. | UI updates optimistically; on success the affected React Query cache is invalidated and re-renders consistent data; on failure it rolls back and a toast error appears. Never local `useState` + direct service calls. |
| GLB-007 | Toast | Any success/error mutation | 1. Trigger a success, then an error. | A react-hot-toast appears with specific copy, is dismissible, auto-dismisses; errors show the real reason, not a generic stack or `[object Object]`. |
| GLB-008 | Deep-link | Deep routes (grant overview, application by reference, report by id, AI workspace by slug, /project/[id]/updates, impact-criteria, portfolio-report preview, milestones/:projectId, oauth/consent) | 1. Open a deep URL in a fresh tab. 2. Use browser Back then Forward. | Page loads standalone (no dependency on prior in-app nav); Back/Forward restore prior view + scroll without crashing or duplicate fetch loops. |
| GLB-009 | SSR/Hydration | SSR'd pages — sweep: `/`, `/communities`, `/community/[id]/projects`, `/project/[id]`, `/funders`, `/knowledge`, `/dashboard`, `/mcp/connect`, `/seeds`, `/oauth/consent`, `/funding-map`, apply/donate/ask-karma/evaluate, all `(whitelabel)` community surfaces | 1. View source / load with JS slow. 2. Open console. | No React hydration-mismatch warnings/errors; server markup matches client; JSON-LD present in SSR HTML where expected; navbar/footer/theme stable server→client; no layout shift. |
| GLB-010 | Web3 (disconnected) | Any on-chain action | 1. Disconnect wallet. 2. Attempt the action. | Action blocked or prompts connect/login; no unhandled promise rejection; clear "Wallet Disconnected" message. |
| GLB-011 | Web3 (wrong network) | Wallet on unsupported chain | 1. Switch to wrong chain. 2. Attempt an on-chain action. | App prompts/auto-triggers chain switch ("Please switch to the correct network"); action proceeds only after switch; cancel leaves state clean. |
| GLB-012 | Web3 (reject) | Any on-chain action | 1. Start the action. 2. Reject the signature/tx. | Non-fatal "Transaction Rejected / You rejected the transaction request." message; UI returns to actionable state; no stuck spinner. |
| GLB-013 | Web3 (insufficient gas) | Wallet with ~0 gas | 1. Attempt an on-chain action. | "Insufficient Funds" message; no crash; retry possible after funding. |
| GLB-014 | Web3 (pending tx) | A tx already pending | 1. Trigger a second wallet request. | "Request Pending — A wallet request is already pending" message; app does not double-submit. |
| GLB-015 | Web3 (Privy↔Wagmi race) | Just-logged-in via Privy, wagmi not yet hydrated | 1. Log in, immediately (~1s) trigger an on-chain/owner-gated control. | App tolerates the brief "authenticated-but-disconnected" window: control disabled/loading rather than throwing; enables once wagmi connects; RBAC fallbacks (`isStaff ?? false`) do not flash forbidden content; navbar never flashes "Sign in" or triggers a logout. |

| GLB-018 | Destructive | Any delete/revoke/regenerate/pause (grant delete, milestone delete, completion revoke, application withdraw, API key revoke, prompt delete, work-board task delete, chain payout address change, portfolio-report config delete, faucet pause/resume) | 1. | GLB-021 | A11y (SR labels) | Icon-only buttons (cart remove, share, edit, delete, attach, copy) and form fields | 1. Navigate with a screen reader. | Every icon-only button has an accessible name; every input has an associated label; required fields announce required state. |
| GLB-022 | Console cleanliness | Any route | 1. Exercise flows with console open. | No `console.log/debug/warn` from app code; no uncaught errors; no React key warnings. (Explicitly verify `useMilestone.ts` output-save failures surface a user-visible warning AND leave no committed `console.error` — see GMI-064.) |
| GLB-023 | A11y (`.catch()` sweep) | Every copy-to-clipboard + async browser API (PUB-027 wallet address, PUB-095 share, GMI-463 API key, GMI-472 connect config, ADM browse-application links, `/api/geo` geolocation) | 1. Force the clipboard/geolocation API to fail. | Every async browser API has a `.catch()`: copy failure shows a toast, no unhandled rejection; geolocation failure degrades gracefully. |
