# gap-app-v2 — On-Chain / Signature-Gated QA Plan (Deferred Cases)

This plan covers every flow that **broadcasts a blockchain transaction or requires a wallet signature** — the cases that were intentionally **DEFERRED** in the manual regression run (`qa-run-report.md`) because no funded wallet was available and the runner was instructed to stop at the signature gate. Run this **after** a funded testnet wallet is provisioned. It is grounded in the actual transaction code (hooks/services/SDK calls cited per flow).

> **Last generated:** Generated by QA workflow. **Scope:** end-to-end on-chain behavior only — for UI/validation up to the signature gate, see `manual-regression-plan.md`.

---

## How to use this plan

1. Execute every case by hand in a real browser with the wallet(s) described in **Test environment**. Mark each ID **Pass / Fail / Blocked**, record the **tx hash + explorer link + chain** for every broadcast, and screenshot failures.
2. **Testnet only. Small amounts.** Never run these against mainnet or with significant funds. Every amount should be the minimum the contract accepts.
3. Run each on-chain flow **twice**: once with an **external wallet** (MetaMask) and once with a **Privy embedded / gasless smart account** (social login). The signing path differs (`setupChainAndWallet()` resolves wagmi external wallet → Privy wallet → smart account) and is the highest-risk surface — see flow **OCH-GASLESS**.
4. **Path type** legend: `Happy` (signs + confirms), `Reject` (user rejects signature/tx in wallet), `Funds` (insufficient gas/token/fee), `Chain` (wrong network / chain-switch), `Revert` (tx reverts on-chain), `Index` (post-tx indexer/polling lag), `Perm` (role/eligibility gate), `Boundary` (expiry, nonce, max/min, multi-chain batch).

## Global cross-cutting on-chain checklist (applies to EVERY flow below)

| ID | Path type | What to verify on every signature-gated action |
|----|-----------|--------------------------------------------------|
| OCH-G01 | Reject | Rejecting the signature/tx in the wallet shows a user-facing error toast (not a silent hang or a blank), the stepper/modal returns to an actionable state, and the form data is preserved (no data loss). |
| OCH-G02 | Funds | With insufficient gas/native balance, the wallet's failure surfaces as a readable error; the app does not mark the action successful. |
| OCH-G03 | Chain | If the wallet is on the wrong network, the app detects it (donations call `walletClient.getChainId()`, not just wagmi state), prompts/performs a chain switch, waits, and re-verifies before sending. Cancelling the switch aborts cleanly. |
| OCH-G04 | Revert | If the tx reverts on-chain, the receipt-status check fails the action with an error (no false success); transfers/optimistic state roll back. |
| OCH-G05 | Index | After a successful tx, the attestation/record appears once the indexer catches up; the polling loop (≈1000×1.5s for attestations) shows progress and, if the indexer lags past timeout, fails gracefully rather than hanging forever. |
| OCH-G06 | Boundary | The post-login Privy↔Wagmi race (dashboard flicker) does not cause a double-submit or a dropped click on the sign button — wait for the surface to settle before signing. |
| OCH-G07 | Happy | After success: correct success toast, query invalidation refreshes the UI without a manual reload, and any share/next-step dialog appears. Console has no uncaught errors. |

## 1. Attestation flows (EAS via karma-gap-sdk)

Common code: `setupChainAndWallet()` resolves the signer; success path POSTs the tx hash to `INDEXER.ATTESTATION_LISTENER()` then polls until indexed. Failures call `showError(...)` + `errorManager()` (Sentry) and keep the form open.

### OCH-PROJECT — Create / update project (`components/Dialogs/ProjectDialog/index.tsx`)
| ID | Path type | Precondition | Steps | Expected result |
|----|-----------|--------------|-------|-----------------|
| OCH-PRJ-01 | Happy | W1 logged in, on a gapSupported testnet | Open Create Project, fill all required fields, pick the testnet chain, submit, **sign** the Project+ProjectDetails+MemberOf attestations | Stepper runs preparing→indexing→indexed; redirects to the new project; project resolves by slug (the `checkSlugExists` poll succeeds). |
| OCH-PRJ-02 | Reject | same | At the signature prompt, reject in wallet | `MESSAGES.PROJECT.CREATE.ERROR` toast; dialog stays open with data intact; logged to Sentry. |
| OCH-PRJ-03 | Chain | W1 on a non-supported / different chain | Submit | App switches to the selected chain (or blocks with a clear message); does not silently no-op (note: code path can `return` silently if wallet setup fails — verify the user sees *something*). |
| OCH-PRJ-04 | Index | Indexer paused/slow | Submit + sign | Polling shows progress; on prolonged lag the UI does not hang forever and the project still appears once indexing completes. |
| OCH-PRJ-05 | Happy | W1 owns a project | Edit the project, change details, sign | Old ProjectDetails revoked + new attested; `MESSAGES.PROJECT.UPDATE` success; changes visible after indexing. |
| OCH-PRJ-06 | Perm | W2 (not owner) | Attempt edit on W1's project | Edit action hidden/blocked; if forced, attestation is rejected by the contract/owner check → error, no change. |

## 2. Donations — Permit2 batch (`hooks/useDonationTransfer.ts`, `components/DonationCheckout.tsx`)

Two-step: ERC-20 **approve → Permit2** (only for tokens), then **`batchDonateWithPermit`** (tokens, EIP-712 signature) or **`batchDonate`** (native value). Phase machine: checking→approving→donating→completed/error. Pre-flight `validatePayments()` checks amount>0, decimals 0–18, and per-token balance.

| ID | Path type | Precondition | Steps | Expected result |
|----|-----------|--------------|-------|-----------------|
| OCH-DON-01 | Happy | W1 funded with a **native** test token; a project with a payout address on this chain | Add 1 grant to cart, enter amount, checkout, confirm `batchDonate` tx | Phase donating→completed; receipt status success; success state per transfer; UI refreshes. |
| OCH-DON-02 | Happy | W1 funded with an **ERC-20** test token | Donate with the ERC-20 | First an **approve-to-Permit2** tx, then a **Permit2 typed-data signature**, then `batchDonateWithPermit`; approval-progress callback advances; success. |
| OCH-DON-03 | Reject | OCH-DON-02 setup | Reject the **approval** tx, then (separately) reject the **Permit2 signature** | Each rejection is sanitized via `parseDonationError`, phase → error, `isExecuting` resets; cart preserved; retry works. |
| OCH-DON-04 | Funds | amount > wallet balance | Attempt checkout | `validatePayments` blocks pre-flight with a per-token balance error; no tx sent. |
| OCH-DON-05 | Chain | wallet on wrong chain | Checkout | App reads real provider chain via `getChainId()`, switches, waits 1.5s, re-verifies; if it can't match, throws a clear chain-mismatch error (no wrong-chain send). |
| OCH-DON-06 | Revert | contract reverts (e.g. transfer fails) | Force a failing transfer | Receipt status ≠ success → each transfer marked error, error re-thrown, user sees failure (no false "donated"). |
| OCH-DON-07 | Boundary | multi-chain cart | Add grants on 2 chains | Payments grouped by chainId; per-chain approval+donation; verify both chains complete and totals are correct. |
| OCH-DON-08 | Boundary | Permit2 deadline | Sit on the checkout > deadline (now+3600s) then submit | Expired permit is rejected by the contract; app surfaces an error and lets the user re-sign. |
| OCH-DON-09 | Funds | no payout address configured for the project/chain | Attempt donate | Blocked with guidance (donation needs a configured payout address). |

## On-chain smoke test (quick go/no-go with a funded wallet)

Run in order; if any breaks, stop and triage before the full pass:
1. **OCH-GAS-01** — social-login user creates a project (gasless attestation succeeds).
2. **OCH-MS-02** — complete a milestone with an external wallet (attestation + indexer poll).
3. **OCH-DON-02** — donate an ERC-20 (approve → Permit2 sign → batch donate).
4. **OCH-PAY-01** — propose a Safe disbursement (owner path).
5. **OCH-CLM-01** — direct Hedgey claim.
6. **OCH-G01 / G03 / G04** — on any one of the above, verify reject, chain-switch, and revert all produce clean errors (no false success, no blank).

## Appendix — open questions / needs-info (confirm before running)

- **Which testnet** does the target GAP indexer index? Attestations only "complete" against the indexer that watches that chain — pick the test wallet's chain accordingly.
- **Filecoin specifics:** confirm whether any filecoin-community flow actually runs on FVM (FIL gas) or attests on an L2; provision the matching faucet only if needed.
- **Endorsement attestation** (OCH-IMP-02): the exact hook/schema wasn't pinned in the inventory — confirm the entry point and reviewer permission before testing.
- **Faucet send** internals (OCH-FCT-02): confirm whether it's an admin wallet tx vs a backend-signed transfer, and which chains it funds.
- **Safe test fixtures:** you must pre-create a deployed+indexed Safe with known owner/delegate sets on a Safe-service testnet; this is the longest setup item.
- **Hedgey campaign:** you must have a campaign where the test wallet is eligible with a valid proof and the claim fee funded.
